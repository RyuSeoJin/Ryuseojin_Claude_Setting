# 역기획서 품질 체크리스트

> 3단계 검증으로 역기획서의 완성도를 평가한다.

---

## Level 1: 필수 완성도 (Must Have)

모든 항목을 통과해야 "역기획서"로 인정된다.

### 문서 존재 여부

| # | 문서 | 확인 |
|---|------|------|
| 1 | 정의서 | [ ] |
| 2 | 구조도 (Mermaid 다이어그램) | [ ] |
| 3 | 플로우차트 (Mermaid 다이어그램) | [ ] |
| 4 | 상세 명세서 | [ ] |
| 5 | 데이터 테이블 | [ ] |
| 6 | 예외 처리 명세 | [ ] |

### 정의서 필수 항목

- [ ] 시스템명이 명시되어 있다
- [ ] 한 줄 정의가 있다
- [ ] 핵심 목적이 유저 관점 + 사업 관점으로 분리되어 있다
- [ ] 용어 정의 표가 3개 이상의 용어를 포함한다
- [ ] 분석 범위와 범위 밖이 명시되어 있다
- [ ] 관련 시스템이 1개 이상 나열되어 있다

### 플로우차트 필수 항목

- [ ] Mermaid 문법으로 작성되었다
- [ ] 시작점과 종료점이 있다
- [ ] 조건 분기가 1개 이상 있다
- [ ] 유저 액션과 시스템 처리가 구분되어 있다

### 데이터 테이블 필수 항목

- [ ] 최소 1개 테이블이 존재한다
- [ ] 컬럼 이름, 타입, 설명이 있다
- [ ] 샘플 데이터가 3행 이상 있다

### 예외 처리 필수 항목

- [ ] 엣지 케이스가 3개 이상 있다
- [ ] 각 예외에 "조건 → 시스템 반응 → 유저 피드백" 구조가 있다

### Chart.js 렌더링

- [ ] Chart.js CDN이 포함되어 있다
- [ ] 최소 1개의 Chart.js 차트가 존재한다
- [ ] 차트 canvas에 유효한 ID가 있다
- [ ] Chart.js 초기화 스크립트가 있다

---

## Level 2: 품질 권장 (Should Have)

"이 문서만으로 구현 가능한가?" 수준의 품질을 확보한다.

### 구현 가능성 테스트

- [ ] 정의서만 읽고 시스템의 전체 윤곽을 이해할 수 있다
- [ ] 플로우차트만 보고 유저 행동 시나리오를 재현할 수 있다
- [ ] 데이터 테이블만 보고 DB 스키마를 설계할 수 있다
- [ ] 상세 명세서만 보고 UI를 구현할 수 있다
- [ ] 예외 처리만 보고 에러 핸들링을 구현할 수 있다

### 용어 일관성

- [ ] 정의서의 용어 정의 표에 있는 용어가 전체 문서에서 일관되게 사용된다
- [ ] 동일 개념에 2개 이상의 용어를 사용하지 않는다
- [ ] 약어 사용 시 첫 등장에서 풀네임을 표기한다

### 수치 근거

- [ ] 모든 확률, 비용, 보상 수치에 출처가 있다 (직접 측정 / 공식 정보 / [추정])
- [ ] [추정]으로 표기된 수치에 추정 근거가 있다
- [ ] 수치 간 모순이 없다 (예: 합산 확률이 100%를 초과하지 않음)

### 다이어그램 품질

- [ ] 구조도 노드가 15개 이하이다
- [ ] 플로우차트 노드가 15개 이하이다 (초과 시 서브 플로우로 분리)
- [ ] 모든 노드에 한국어 라벨이 있다
- [ ] 관계선에 동사형 라벨이 있다
- [ ] 범례가 포함되어 있다

### 상세 명세 품질

- [ ] UI 레이아웃이 영역별로 분해되어 있다
- [ ] 인터랙션에 입력 방식(탭/스와이프/롱프레스 등)이 명시되어 있다
- [ ] 애니메이션에 타이밍(초 단위)이 명시되어 있다
- [ ] 상태 전이에 모든 상태 쌍의 전이 가능 여부가 정의되어 있다

### 웹 리서치 완성도

- [ ] 5개 리서치 카테고리 중 3개 이상 수집되었다
- [ ] 모든 데이터에 신뢰도 등급이 표기되어 있다
- [ ] 출처 URL이 기록되어 있다
- [ ] [추정] 데이터에 근거가 있다

### 수치 분석 깊이

- [ ] 4대 영역 중 해당 영역이 1개 이상 분석되었다
- [ ] 분석에 Chart.js 시각화가 포함되어 있다
- [ ] 수식/공식이 명시되어 있다
- [ ] 분석 결과에 해석/인사이트가 있다

## Level 3: 포트폴리오 가점 (Nice to Have)

포트폴리오로 제출할 때 차별화 요소.

### 비교 분석

- [ ] 동일 시스템을 가진 다른 게임 1개 이상과 비교 분석이 있다
- [ ] 비교 매트릭스(표)가 있다
- [ ] 각 게임의 트레이드오프가 분석되어 있다
- [ ] 비교를 통한 인사이트가 도출되어 있다

### 개선안 제시

- [ ] 현재 시스템의 문제점이나 개선 가능성을 1개 이상 제시한다
- [ ] 개선안에 구체적인 스펙이 있다 (단순 의견이 아님)
- [ ] 개선안의 기대 효과를 수치로 예측한다

### 장르 커스터마이징

- [ ] 장르가 감지/선택되어 있다
- [ ] 장르별 추가 섹션이 포함되어 있다
- [ ] 장르별 전용 Chart.js 차트가 있다

### 레이아웃 품질

- [ ] 문서 간 스타일이 일관된다 (제목 크기, 표 형식 등)
- [ ] 목차가 있다
- [ ] 페이지 번호 / 섹션 번호가 있다
- [ ] PPT 자료가 포함되어 있다
