# 웹 리서치 방법론 가이드

> web-researcher 에이전트가 게임 시스템 역기획에 필요한 데이터를 수집하는 방법론을 정의한다.

---

## 검색 쿼리 전략

### 카테고리 1: 기본 정보 (3개 쿼리)

게임과 시스템의 공식적 사실 정보를 수집한다.

| # | 쿼리 템플릿 | 목적 | 기대 소스 |
|---|-----------|------|----------|
| Q1 | `"{게임명} {시스템명} wiki"` | 공식 위키 기반 시스템 개요 | Fandom, 나무위키, 공식위키 |
| Q2 | `"{게임명} {시스템명} mechanics guide"` | 시스템 상세 메커니즘 가이드 | 공략 사이트, 유저 가이드 |
| Q3 | `"{게임명} patch notes {시스템명}"` | 시스템 변경 이력 | 공식 사이트, 스팀 뉴스 |

**Q1 활용법:**
- 위키 문서에서 시스템의 기본 정의, 용어, 규칙을 추출
- 위키 문서의 "See Also" / "관련 문서" 링크를 통해 연관 시스템 파악
- 문서 하단의 출처/참고를 통해 공식 소스 역추적

**Q2 활용법:**
- 가이드 문서에서 숨겨진 메커니즘, 팁, 계산 공식 추출
- 스크린샷/영상 첨부된 가이드에서 UI 구조 파악
- "초보자 가이드" vs "상급자 가이드"를 모두 탐색하여 깊이 확보

**Q3 활용법:**
- 패치노트에서 시스템의 변경 이력을 시간순으로 정리
- 너프/버프 이력으로 밸런싱 의도 파악
- 최신 패치 기준으로 현재 상태 확정

### 카테고리 2: 커뮤니티 반응 (2개 쿼리)

유저 커뮤니티의 분석과 의견을 수집한다.

| # | 쿼리 템플릿 | 목적 | 기대 소스 |
|---|-----------|------|----------|
| Q4 | `"{게임명} {시스템명} reddit analysis"` | 커뮤니티 심층 분석 | Reddit 서브레딧 |
| Q5 | `"{게임명} {시스템명} tier list"` | 유저 평가 기반 우선순위 | 공략 사이트, Reddit |

**Q4 활용법:**
- Reddit에서 "Math" / "Analysis" / "Guide" 태그 게시글 우선 탐색
- 댓글의 반론/보충 정보도 함께 수집 (다각적 관점)
- 데이터마이닝 결과 공유 게시글 확인

**Q5 활용법:**
- 티어 리스트에서 시스템 내 우선순위/가치 판단 기준 추출
- 복수의 티어 리스트 비교하여 합의점 도출
- 티어 변동 이력으로 메타 변화 파악

### 카테고리 3: 수익 데이터 (2개 쿼리)

게임의 비즈니스 모델과 수익 구조를 파악한다.

| # | 쿼리 템플릿 | 목적 | 기대 소스 |
|---|-----------|------|----------|
| Q6 | `"{게임명} revenue sensor tower"` | 매출 데이터 | Sensor Tower, data.ai |
| Q7 | `"{게임명} monetization model analysis"` | 수익화 모델 분석 | 게임 산업 미디어 |

**Q6 활용법:**
- 월별/분기별 매출 추이로 시스템 변경과 매출 상관관계 파악
- 국가별 매출 비교로 지역별 특성 이해
- 경쟁작 대비 매출 포지셔닝 확인

**Q7 활용법:**
- 수익화 모델 유형 분류 (F2P, 구독, 배틀패스, 가챠 등)
- 과금 유도 지점과 시스템의 연결고리 파악
- 업계 분석가의 평가 수집

### 카테고리 4: 업데이트 이력 (1개 쿼리)

시스템의 진화 과정을 추적한다.

| # | 쿼리 템플릿 | 목적 | 기대 소스 |
|---|-----------|------|----------|
| Q8 | `"{게임명} {시스템명} changelog history"` | 변경 이력 전체 | 공식 사이트, 위키 |

**Q8 활용법:**
- 시스템 최초 도입 시점과 초기 설계 파악
- 주요 변경점을 타임라인으로 정리
- 변경 빈도와 규모로 시스템의 성숙도 판단
- 롤백된 변경사항으로 실패한 설계 의도 파악

### 카테고리 5: 경쟁작 정보 (2개 쿼리)

유사 시스템을 가진 다른 게임과 비교한다.

| # | 쿼리 템플릿 | 목적 | 기대 소스 |
|---|-----------|------|----------|
| Q9 | `"games similar to {게임명} {시스템명}"` | 유사 시스템 가진 게임 발견 | 게임 미디어, 커뮤니티 |
| Q10 | `"{게임명} vs {경쟁게임} {시스템명} comparison"` | 직접 비교 분석 | YouTube, 블로그, Reddit |

**Q9 활용법:**
- 동일 장르 내 유사 시스템 구현 사례 수집 (2~3개)
- 시스템명이 다르더라도 동일한 기능의 시스템 식별
- "영향을 받은 게임" / "영향을 준 게임" 관계 파악

**Q10 활용법:**
- 직접 비교 콘텐츠에서 차별점/공통점 추출
- 유저 선호도 기반 장단점 파악
- 경쟁작의 개선 사항 중 해당 게임에 적용 가능한 인사이트

### 쿼리 실행 순서

```
1차: Q1 (위키) → 기본 정보 확보, 용어 파악
2차: Q2 (가이드) + Q3 (패치노트) → 상세 메커니즘 + 변경 이력
3차: Q4 (Reddit) + Q5 (티어) → 커뮤니티 관점
4차: Q6 (매출) + Q7 (수익화) → 비즈니스 관점
5차: Q8 (변경이력) → 시간축 데이터 보충
6차: Q9 (유사게임) + Q10 (비교) → 경쟁 관점
```

각 단계에서 얻은 키워드/용어를 다음 단계 쿼리에 반영하여 정확도를 높인다.

## 신뢰도 등급 시스템

모든 수집 데이터에 신뢰도 등급을 부여한다.

### 등급 정의

| 등급 | 소스 유형 | 신뢰 점수 | 본문 표기 | 설명 |
|------|----------|----------|----------|------|
| **A** | 공식 문서 / 패치노트 / 인게임 표시 | 5/5 | 그대로 사용 | 개발사가 직접 공개한 정보 |
| **B** | 커뮤니티 위키 (나무위키, Fandom, GameWith) | 4/5 | 그대로 사용 | 다수가 검증한 위키 정보 |
| **C** | 유저 분석 / 가이드 (검증된 작성자) | 3/5 | 그대로 사용 | 개인이 분석했으나 데이터 기반 |
| **D** | Reddit/포럼 댓글 (일반 유저) | 2/5 | 그대로 사용 | 검증 안 된 일반 의견 |
| **E** | 추정 / 역산 / 간접 추론 | 1/5 | **[추정]** 필수 표기 | 직접 확인 불가, 추론에 의존 |

### 등급 적용 규칙

1. **동일 데이터에 복수 소스가 있으면**: 가장 높은 등급 적용
2. **소스 간 수치가 다르면**: 양쪽 모두 기록 + 높은 등급 소스 채택
3. **E등급 데이터만 있으면**: 반드시 `[추정]` 표기 + 추정 근거 기술
4. **등급 판정이 애매하면**: 한 단계 낮은 등급 적용 (보수적 판정)

### 등급별 데이터 처리

```
A등급: 그대로 사용. 출처 URL만 기록.
B등급: 그대로 사용. 최종 수정일 확인 (오래된 위키 주의).
C등급: 그대로 사용. 분석 방법론이 타당한지 간략 평가.
D등급: 다른 소스로 교차 검증 시도. 실패 시 그대로 사용.
E등급: [추정] 표기 필수. 추정 근거와 오차 범위 기술.
```

## 폴백 전략

검색 결과가 부족하거나 WebFetch가 실패할 때의 대안 경로.

### 폴백 체인 1: 검색 언어 전환

```
한글 검색 실패 → 영문 검색 시도
예: "원신 원소반응" → "Genshin Impact elemental reactions"

영문 검색 실패 → 일본어/중국어 키워드 시도
예: "原神 元素反応" (일본어), "原神 元素反应" (중국어)
```

### 폴백 체인 2: 게임명 변형

```
정식 명칭 실패 → 약어/줄임말 시도
예: "메이플스토리" → "메이플", "MapleStory" → "Maple"

약어 실패 → 별명/커뮤니티 명칭 시도
예: "블루 아카이브" → "블아", "Blue Archive" → "BA"
```

### 폴백 체인 3: 소스 사이트 순회

```
1순위: 공식 위키 / 공식 사이트
2순위: 나무위키 (namu.wiki)
3순위: Fandom Wiki ({game}.fandom.com)
4순위: GameWith (gamewith.jp) - 일본 게임의 경우
5순위: Honey Hunter / 기타 전문 DB 사이트
6순위: YouTube 검색 (데이터 수집/분석 영상)
7순위: 커뮤니티 데이터마이닝 사이트 (GitHub 등)
```

### 폴백 체인 4: 데이터 유형별 대안

| 필요 데이터 | 1차 소스 | 2차 소스 | 3차 소스 (최후) |
|------------|---------|---------|----------------|
| 확률 데이터 | 인게임 표시/공식 공지 | 커뮤니티 실험 결과 | 유사 게임 기반 추정 |
| 비용 데이터 | 인게임 확인/위키 | 공략 사이트 | 유저 후기/영상 캡처 |
| 공식 데이터 | 공식 문서/데이터마이닝 | Reddit 분석글 | 역산/추정 |
| 업데이트 이력 | 공식 패치노트 | 위키 변경 이력 | 커뮤니티 기억/정리글 |
| 매출 데이터 | Sensor Tower / data.ai | 게임 미디어 보도 | 생략 (필수 아님) |

### 폴백 판단 기준

- **쿼리당 최대 재시도**: 3회 (원본 + 변형 2회)
- **카테고리당 최소 시간**: 결과 0건이면 즉시 폴백
- **전체 리서치 포기 기준**: 카테고리 1(기본 정보)마저 데이터 수집 불가 시 유저에게 알림

## 결과 구조화 규칙

수집한 모든 데이터를 일관된 형식으로 정리한다.

### 규칙 1: 출처 기록 필수

모든 데이터 항목에 출처 URL을 기록한다.

```markdown
| 데이터 항목 | 값 | 출처 | 등급 |
|------------|-----|------|------|
| SSR 확률 | 1.6% | https://example.com/gacha | A |
| 천장 값 | 90회 | https://reddit.com/r/example | C |
```

### 규칙 2: 수치 데이터 별도 표 정리

수치 데이터는 반드시 별도 표로 정리하여 data-analyst에게 전달하기 쉽게 한다.

```markdown
### 확률 데이터
| 등급 | 기본 확률 | 천장 적용 확률 | 출처 |
|------|----------|--------------|------|
| SSR | 0.6% | 소프트천장 74회부터 증가 | [A] 공식 |
| SR | 5.1% | - | [A] 공식 |
| R | 94.3% | - | [A] 공식 |
```

### 규칙 3: 날짜/버전 정보 필수

```markdown
- 데이터 기준 날짜: 2025-03-15
- 게임 버전: v4.5.0
- 해당 패치 날짜: 2025-03-12
```

### 규칙 4: 모순 정보 처리

```markdown
### 모순 데이터: 강화 성공률
- 소스 A (공식, 등급 A): +15 강화 성공률 30%
- 소스 B (위키, 등급 B): +15 강화 성공률 25%
- **채택**: 소스 A (등급 A, 최신 패치 기준)
- **비고**: 소스 B는 패치 전 데이터일 가능성
```

### 규칙 5: 누락 항목 명시

```markdown
### 수집 실패 항목
- [ ] 정확한 데미지 공식 (공식 비공개, E등급 추정만 가능)
- [ ] 일본 서버 전용 이벤트 데이터 (접근 불가)
- [ ] 서버별 매출 비교 (유료 데이터)
```

## 리서치 브리프 작성 가이드

### 필수 섹션 (8개)

| # | 섹션 | 최소 요구사항 | 소스 쿼리 |
|---|------|-------------|----------|
| 1 | 게임 개요 | 장르, 개발사, 출시일, 플랫폼 | Q1 |
| 2 | 시스템 정의 | 한 줄 정의, 핵심 기능 3개 이상 | Q1, Q2 |
| 3 | 시스템 메커니즘 | 동작 원리, 규칙, 조건 상세 | Q2, Q3 |
| 4 | 수치 데이터 | 확률/비용/레벨 데이터 (표 형식) | Q1, Q2, Q4 |
| 5 | 업데이트 이력 | 주요 변경점 3개 이상 (날짜 포함) | Q3, Q8 |
| 6 | 커뮤니티 반응 | 긍정/부정 의견 각 2개 이상 | Q4, Q5 |
| 7 | 수익화 연결 | 시스템과 과금의 관계 | Q6, Q7 |
| 8 | 경쟁작 비교 | 유사 게임 2개 이상 | Q9, Q10 |

### 리서치 완성도 자가 체크리스트

```markdown
## 리서치 완성도 체크

### 필수 (전체 통과 필요)
- [ ] 게임명 + 시스템명이 정확히 명시됨
- [ ] 시스템 한 줄 정의가 작성됨
- [ ] 최소 3개 카테고리에서 데이터 수집됨
- [ ] 수치 데이터가 1개 이상 표로 정리됨
- [ ] 모든 데이터에 출처 URL이 기록됨

### 권장 (가능한 한 통과)
- [ ] 5개 카테고리 전체에서 데이터 수집됨
- [ ] 신뢰도 등급이 모든 데이터에 부여됨
- [ ] 모순 데이터가 명시적으로 처리됨
- [ ] 수집 실패 항목이 명시됨
- [ ] 경쟁작 2개 이상 비교됨
- [ ] 업데이트 이력이 3개 이상 정리됨
- [ ] 커뮤니티 긍정/부정 의견이 균형 있게 수집됨

### 산출물 형식
- [ ] web-research-brief-template.md 형식을 준수함
- [ ] 수치 데이터가 data-analyst 전달용으로 별도 정리됨
- [ ] 날짜/버전 정보가 포함됨
```

### 리서치 소요 시간 가이드

| 단계 | 예상 소요 | 비고 |
|------|----------|------|
| 1차 검색 (Q1~Q3) | 가장 먼저 | 기본 정보 확보 우선 |
| 2차 검색 (Q4~Q5) | 1차 직후 | 커뮤니티 데이터 |
| 3차 검색 (Q6~Q8) | 2차 직후 | 비즈니스 + 이력 |
| 4차 검색 (Q9~Q10) | 3차 직후 | 경쟁작 (선택적) |
| 결과 구조화 | 검색 완료 후 | 브리프 작성 |

## WebFetch 사용 가이드

### 유효한 URL 패턴

```
공식 사이트:    https://{game}.{company}.com/news/...
나무위키:       https://namu.wiki/w/{게임명}/{시스템명}
Fandom:        https://{game}.fandom.com/wiki/{SystemName}
GameWith:      https://gamewith.jp/{game}/article/show/{id}
Reddit:        https://www.reddit.com/r/{subreddit}/comments/{id}/...
```

### WebFetch 프롬프트 템플릿

```
"이 페이지에서 다음 정보를 추출해주세요:
1. {시스템명}의 핵심 메커니즘 (동작 원리)
2. 관련 수치 데이터 (확률, 비용, 레벨 등)
3. 최근 변경사항 (패치 이력)
4. 유저 반응/평가
가능한 한 구체적인 수치와 함께 정리해주세요."
```

### WebFetch 실패 시 대응

```
HTTP 403/429 → 다음 소스로 이동 (폴백 체인 3)
HTTP 404     → URL 변형 시도 (경로 변경, 언어 코드 변경)
타임아웃      → 재시도 1회 후 다음 소스로 이동
콘텐츠 없음   → 검색 결과 스니펫으로 대체
```
