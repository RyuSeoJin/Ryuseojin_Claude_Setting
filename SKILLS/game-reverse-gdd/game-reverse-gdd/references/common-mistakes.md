# 역기획서 흔한 실수 & 방지법

> 역기획서 작성 시 빈번하게 발생하는 8가지 실수와 구체적인 해결 전략.

---

## 실수 1: 범위 과대 설정

**문제**: 시스템 전체를 한 문서에 담으려다 모든 내용이 얕아진다.

**Bad:**
> "원신의 전투 시스템 역기획서"
> → 원소 반응, 스킬, 타겟팅, 데미지 계산, 버프/디버프를 모두 다루려다 각각 1페이지

**Good:**
> "원신의 원소 반응 데미지 계산 시스템 역기획서"
> → 원소 반응 타입별 계산 공식, 레벨 보정, 원소 마스터리 영향을 심층 분석

**해결 전략:**
- Step 1에서 MECE 원칙으로 범위를 검증한다
- "이 시스템을 하나의 팝업/화면에서 설명할 수 있는가?" 테스트
- 범위가 넓으면 하위 시스템 2~3개로 분리 후 하나를 선택

---

## 실수 2: 분석문/감상문 작성

**문제**: 기획서가 아닌 리뷰어의 감상을 적는다.

**Bad:**
> "이 시스템은 유저에게 큰 재미를 주며 매우 잘 설계되었다. 특히 뽑기의 연출이 화려하여 유저의 기대감을 높인다."

**Good:**
> "뽑기 연출 시퀀스: 탭 → 0.5초 페이드 → 3초 연출 (별 등급별 이펙트 차등) → 결과 팝업. 최고 등급 연출 시 화면 전체 플래시 + 전용 BGM 전환 (BGM_gacha_ssr.ogg)"

**해결 전략:**
- 모든 문장에서 "재미있다", "잘 만들었다" 등 주관적 표현을 제거한다
- "개발자가 이 문장으로 무엇을 구현할 수 있는가?" 자가 점검
- 감상은 Step 3 설계 의도 분석에서만 허용 (근거 필수)

---

## 실수 3: 스크린샷 나열

**문제**: 스크린샷을 붙여놓고 설명 없이 "참고"라고만 적는다.

**Bad:**
> [스크린샷 1] 메인 화면
> [스크린샷 2] 상점 화면
> [스크린샷 3] 결과 화면

**Good:**
> **상점 화면 구성:**
> - 상단: 보유 재화 표시 (다이아 / 골드 / 티켓)
> - 중앙: 상품 그리드 (3x4, 스크롤 가능)
> - 각 상품 카드: 아이콘 + 이름 + 가격 + [할인 뱃지(선택)]
> - 하단: 탭 네비게이션 (추천 | 기간한정 | 일반 | 패키지)
> - 탭 전환 시 0.3초 슬라이드 애니메이션

**해결 전략:**
- 스크린샷은 참고 자료일 뿐, 문서의 본문이 아니다
- 모든 스크린샷에 대해 "이 화면에서 무엇이 어디에 어떻게 배치되어 있는지" 텍스트로 기술
- UI 레이아웃은 영역별로 분해하여 명세한다

---

## 실수 4: 소설식 세계관 서술

**문제**: 게임의 스토리나 세계관을 장황하게 서술한다.

**Bad:**
> "태초에 일곱 신이 테이밧 대륙을 지배하였고, 각 신은 원소의 힘을 가졌다. 여행자는 쌍둥이 중 하나로서..."

**Good:**
> **시스템 배경 맥락 (1줄):** 7종 원소(불/물/얼음/번개/바람/풀/바위)가 전투 메카닉의 핵심. 원소 간 반응이 추가 데미지/상태이상을 발생시킴.

**해결 전략:**
- 세계관/스토리는 시스템 이해에 필요한 최소한만 1~2줄로 요약
- "이 정보가 없으면 개발자가 시스템을 구현할 수 없는가?" 기준 적용
- 필요시 용어 정의 표에 세계관 용어를 포함시키는 것으로 대체

---

## 실수 5: 데이터 테이블 누락

**문제**: 시스템의 구동 데이터 없이 기능만 설명한다.

**Bad:**
> "강화 시 성공 확률은 단계가 올라갈수록 낮아진다."

**Good:**

| 강화 단계 | 성공 확률 | 비용(골드) | 실패 시 | 파괴 확률 |
|----------|----------|-----------|---------|----------|
| +1 → +2 | 100% | 1,000 | 유지 | 0% |
| +2 → +3 | 90% | 2,000 | 유지 | 0% |
| +3 → +4 | 80% | 5,000 | -1단계 | 0% |
| +4 → +5 | 60% | 10,000 | -1단계 | 5% |

**해결 전략:**
- 시스템에 관련된 모든 숫자를 테이블로 정리한다
- 확률, 비용, 보상, 쿨타임, 지속시간 등 수치 데이터는 반드시 표로
- 정확한 수치를 모르면 [추정]을 표기하고 추정 근거를 기록

---

## 실수 6: 가독성 무시

**문제**: 구조화 없이 장문의 텍스트를 나열한다.

**Bad:**
> "유저가 상점에 입장하면 상품 목록이 표시되고 상품을 선택하면 상세 정보가 나오며 구매 버튼을 누르면 재화가 차감되고 아이템이 인벤토리에 추가되며 재화가 부족하면 충전 팝업이 뜬다."

**Good:**
> **상점 구매 플로우:**
> 1. 상점 입장 → 상품 목록 표시
> 2. 상품 탭 → 상세 정보 팝업
> 3. [구매] 탭
>    - 재화 충분 → 재화 차감 → 아이템 지급 → 성공 토스트
>    - 재화 부족 → 충전 팝업 표시

**해결 전략:**
- 한 문단에 3줄 이상 연속 서술 금지
- 목록, 표, 다이어그램을 적극 활용
- 조건 분기는 반드시 들여쓰기 + 불릿으로 표현

---

## 실수 7: 예외 처리 무시

**문제**: Happy Path만 기술하고 예외 상황을 다루지 않는다.

**Bad:**
> "유저가 강화 버튼을 누르면 강화가 진행된다."

**Good:**
> **강화 버튼 탭 시:**
> - 재료 & 재화 충분 → 강화 진행
> - 재료 부족 → "재료가 부족합니다" 토스트 + 재료 획득처 바로가기
> - 재화 부족 → 재화 충전 팝업
> - 최대 강화 도달 → 버튼 비활성화 + "최대 강화 달성" 표시
> - 네트워크 오류 → "일시적 오류" 팝업 + 재시도 버튼
> - 강화 중 앱 종료 → 서버 기준 결과 적용 (클라이언트 복귀 시 동기화)

**해결 전략:**
- 모든 유저 액션에 대해 "실패하면?"을 질문한다
- 네트워크, 동시성, 리소스 한계 3가지 관점으로 반드시 체크
- 예외 처리가 3개 미만이면 분석이 부족한 것

---

## 실수 8: 설계 의도 없는 기능 나열

**문제**: 기능을 나열하지만 "왜" 이렇게 설계했는지 분석이 없다.

**Bad:**
> "가챠에는 천장 시스템이 있다. 90회째에 최고 등급이 보장된다."

**Good:**
> "가챠 천장 시스템 (90회 보장):
> - **설계 의도**: 유저의 불만을 관리하면서 최대 결제를 유도
> - **수익화 관점**: 천장까지의 기대 비용 ~$200, BM 핵심 수익원
> - **리텐션 관점**: 천장 카운트가 배너 간 이월되어 장기 플레이 유도
> - **밸런싱 관점**: 천장 보장으로 인한 캐릭터 보유율 상한선 관리"

**해결 전략:**
- Step 3에서 도출한 설계 의도를 각 기능 명세에 연결한다
- 모든 주요 기능에 "수익화/리텐션/UX/밸런싱/소셜" 중 하나 이상의 관점을 태그
- "이 기능이 없으면 어떤 문제가 생기는가?"를 역으로 질문
