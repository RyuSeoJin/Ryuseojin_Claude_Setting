# 10단계 상세 작성 가이드

> 역기획서 작성의 각 단계별 목적, 구체적 작업 지시, 검증 기준을 상세히 기술한다.
> 각 단계의 담당 에이전트가 명시되어 있으며, 에이전트가 자동으로 품질 검증을 수행한다.

---

## Step 1: 대상 게임 & 범위 선정

> **담당**: 유저 + leader 에이전트 협업

### 목적
역기획의 대상 게임과 분석할 시스템의 범위를 MECE 원칙에 따라 명확히 확정한다.

### 범위 선정 기준

**MECE 원칙 적용:**
- **Mutually Exclusive**: 선택한 범위가 다른 시스템과 명확히 구분되는가?
- **Collectively Exhaustive**: 선택한 범위 내의 모든 기능을 다룰 수 있는가?

### 좋은 범위 예시 (10개)

1. "원신의 원소 반응 데미지 계산 시스템"
2. "블루아카이브의 가챠 천장 시스템"
3. "로스트아크의 장비 강화 시스템 (+1~+20)"
4. "쿠키런 킹덤의 왕국 건설 자원 관리"
5. "메이플스토리의 주간 보스 입장/보상 시스템"
6. "브롤스타즈의 트로피 매칭 시스템"
7. "포켓몬고의 친구 친밀도 & 선물 교환"
8. "디아블로4의 파라곤 보드 시스템"
9. "에픽세븐의 장비 세트 효과 시스템"
10. "포트나이트의 배틀패스 보상 트랙 설계"

### 나쁜 범위 예시 (10개)

1. "원신의 전투 시스템" → 너무 넓음. 원소 반응, 스킬, 타겟팅 등 분리 필요
2. "리니지의 모든 시스템" → 게임 전체는 역기획 범위가 아님
3. "RPG 게임의 성장 시스템" → 특정 게임을 지정해야 함
4. "원신 스토리" → 스토리는 시스템이 아님
5. "좋은 게임 UI" → 주관적, 특정 시스템 필요
6. "로스트아크의 PvE 콘텐츠" → 던전, 레이드, 카오스게이트 등 분리 필요
7. "모바일 게임의 수익화" → 특정 게임, 특정 BM 시스템 필요
8. "메이플스토리 전체 강화 시스템" → 스타포스, 잠재능력, 추옵 등 분리 필요
9. "배틀그라운드의 총기" → 총기 목록이 아닌 시스템(사격, 반동, 부착물)으로 정의
10. "리그 오브 레전드" → 게임명만으로는 범위 미확정

### 범위 검증 질문

1. 이 범위를 하나의 팝업/화면에서 설명할 수 있는가?
2. "유저가 [동사]한다"로 한 문장에 표현할 수 있는가?
3. 분석에 2~3시간이면 충분한가?
4. 최종 문서가 15~20페이지에 수렴하는가?
5. 서로 독립적인 2개 이상의 기능이 포함되어 있지 않은가?

### 검증 기준
- [ ] 특정 게임명이 확정되었다
- [ ] 시스템명이 한 줄로 정의 가능하다
- [ ] system-categories.md의 하위 시스템 수준으로 좁혀졌다

## Step 2: 웹 리서치 & 데이터 수집

> **담당**: web-researcher 에이전트 (자동 수집)
> **참조**: `assets/templates/web-research-brief-template.md`, web-research-guide.md

### 목적
WebSearch/WebFetch 도구를 활용하여 역기획에 필요한 정량/정성 데이터를 자동으로 수집한다.

### 웹 리서치 카테고리별 수집 체크리스트

| 카테고리 | 수집 대상 | 우선순위 |
|----------|----------|---------|
| 기본 정보 | 게임 개요, 시스템 공식 설명, 위키 데이터 | 필수 |
| 핵심 메카닉 | 수치 데이터, 공식, 확률 테이블, 밸런스 수치 | 필수 |
| 업데이트 이력 | 패치노트, 변경 히스토리, 너프/버프 이력 | 필수 |
| 커뮤니티 반응 | Reddit, 포럼, 유튜브 리뷰, 커뮤니티 피드백 | 권장 |
| 수익/시장 데이터 | 매출 추정, DAU, 과금 구조 분석 | 권장 |
| 경쟁작 비교 | 유사 시스템 보유 게임, 차별화 포인트 | 권장 |

### 웹 기반 데이터 추출

**WebSearch 쿼리 전략:**

```
# 1단계: 공식 소스 수집 (신뢰도 A)
WebSearch("{게임명}" "{시스템명}" official guide wiki)
WebSearch("{게임명}" patch notes {시스템명})

# 2단계: 수치 데이터 수집 (신뢰도 A-B)
WebSearch("{게임명}" {시스템명} formula calculation data)
WebSearch("{게임명}" {시스템명} probability rate table)

# 3단계: 커뮤니티 데이터 수집 (신뢰도 C-D)
WebSearch("{게임명}" {시스템명} reddit review analysis)
WebSearch("{게임명}" {시스템명} community feedback)

# 4단계: 시장 데이터 수집 (신뢰도 D-E)
WebSearch("{게임명}" revenue sensor tower {연도})
WebSearch("{게임명}" player count DAU statistics)

# 5단계: 경쟁작 수집 (신뢰도 B-C)
WebSearch(games similar to "{게임명}" {시스템명} comparison)
```

**신뢰도 등급 표기 규칙:**
- A: 공식 소스 (개발사 문서, 패치노트)
- B: 공인 커뮤니티 위키 (검증된 데이터)
- C: 커뮤니티 합의 (다수 유저 확인)
- D: 개별 유저 보고 (미검증)
- E: 추정/외삽 (직접 데이터 부재)

### 에이전트 자동 처리

web-researcher 에이전트가 다음을 자동으로 수행한다:
1. 카테고리별 WebSearch 쿼리 실행
2. 검색 결과에서 유효 URL 추출 후 WebFetch로 상세 데이터 수집
3. 수집 데이터에 신뢰도 등급 자동 태깅
4. `web-research-brief-template.md` 형식으로 리서치 브리프 산출물 생성
5. 수집 완성도 자동 평가 (6개 카테고리 중 달성률 계산)

### 검증 기준 (에이전트 자동 검증)
- [ ] 6개 리서치 카테고리 중 3개 이상 수집 완료
- [ ] 모든 데이터에 신뢰도 등급이 표기되어 있다
- [ ] 핵심 수치 데이터가 3개 이상 수집되었다
- [ ] 출처 URL이 기록되어 있다
- [ ] 리서치 브리프 산출물이 생성되었다

## Step 3: 설계 의도 분석

> **담당**: system-analyst 에이전트

### 목적
해당 시스템이 "왜" 이렇게 설계되었는지 5가지 관점에서 심층 분석한다.

### 5가지 분석 관점 & 질문

#### 관점 1: 수익화
1. 이 시스템이 직접 매출을 발생시키는가? (과금 포인트는?)
2. 간접적으로 과금을 유도하는 장치가 있는가?
3. 무과금 유저의 접근 한계는 어디인가?
4. BM과의 연동 구조는 무엇인가?

#### 관점 2: 리텐션
5. 유저가 매일 이 시스템을 사용하게 만드는 장치는?
6. 장기 목표를 제공하는가? (몇 일/주/월 단위?)
7. 이탈 방지 장치가 있는가? (매몰 비용, 진행도 보존 등)
8. 복귀 유저를 위한 장치가 있는가?

#### 관점 3: UX
9. 신규 유저가 이 시스템을 처음 접할 때의 진입 장벽은?
10. 핵심 플로우까지 몇 탭이 필요한가?
11. 정보 과부하를 방지하는 장치가 있는가?
12. 피드백 (시각/청각/촉각)은 충분한가?

#### 관점 4: 밸런싱
13. 성장 속도를 제어하는 장치는 무엇인가? (피로도, 재료 제한 등)
14. Pay-to-Win을 완화하는 장치가 있는가?
15. 인플레이션을 방지하는 재화 소모처가 있는가?
16. 상한선/하한선은 어디에 설정되어 있는가?

#### 관점 5: 소셜
17. 유저 간 비교/경쟁을 유도하는 요소가 있는가?
18. 협력을 유도하는 요소가 있는가?
19. 자랑/공유를 유도하는 요소가 있는가?
20. 커뮤니티에서 이 시스템이 화제가 되는 포인트는?

### 분석 결과 정리 형식

```
| 기능 | 설계 의도 | 관점 | 기대 효과 |
|------|----------|------|----------|
| 천장 시스템(90회) | 불만 관리 + 최대 결제 유도 | 수익화 | 천장까지 약 $200 소비 유도 |
| 천장 카운트 이월 | 장기 플레이 유도 | 리텐션 | 배너 종료 후에도 이탈 방지 |
```

### 검증 기준 (에이전트 자동 검증)
- [ ] 5가지 관점 중 최소 3가지에서 분석이 도출되었다
- [ ] 각 관점에서 최소 2개의 설계 의도가 있다
- [ ] 모든 분석에 근거가 있다 (리서치 브리프 데이터 기반)

## Step 4: 정의서 작성

> **담당**: system-analyst 에이전트

### 목적
시스템의 전체 윤곽을 잡는 기초 문서. 이후 모든 문서의 출발점이 된다.

### 작성 규칙

1. **시스템명**: 게임명 + 시스템명 조합 (예: "원신 원소 반응 데미지 계산 시스템")
2. **한 줄 정의**: 30자 이내로 시스템을 정의 (예: "원소 간 반응으로 추가 데미지를 발생시키는 전투 메카닉")
3. **핵심 목적**: 유저 관점 + 사업 관점으로 분리
   - 유저: "다양한 원소 조합으로 전략적 전투를 즐긴다"
   - 사업: "캐릭터 수집 동기를 부여하여 가챠 매출을 창출한다"
4. **용어 정의**: 시스템 내 모든 전문 용어를 표로 정리
5. **범위**: "이 문서에서 다루는 것"과 "다루지 않는 것"을 명시
6. **관련 시스템**: 의존하는 시스템과 영향을 주는 시스템
7. **대상 유저 행동**: 이 시스템에서 유저가 하는 핵심 행동 3~5개

### 품질 기준
- 정의서만 읽고 시스템의 전체 윤곽을 이해할 수 있는가?
- 용어 정의 없이 이해 불가능한 단어가 본문에 있지 않은가?
- 범위 밖 항목이 명시되어 다른 문서와의 경계가 명확한가?

### 검증 기준 (에이전트 자동 검증)
- [ ] 시스템명이 게임명 + 시스템명 형태이다
- [ ] 한 줄 정의가 30자 이내이다
- [ ] 용어 정의 표에 3개 이상의 용어가 있다
- [ ] 범위 밖 항목이 1개 이상 명시되어 있다
- [ ] 관련 시스템이 1개 이상 나열되어 있다
- [ ] 범위 mindmap이 포함되어 있다

## Step 5: 게임 구조도 작성

> **담당**: system-analyst 에이전트

### 목적
시스템의 구성 요소와 관계를 Mermaid 다이어그램으로 시각화한다.

### 작성 규칙

1. Mermaid `graph TD` 또는 `graph LR` 사용
2. `subgraph`로 시스템 경계 표시
3. 데이터 저장소는 원통형 `[(이름)]`
4. 외부 시스템은 이중 사각형 `[[이름]]`
5. 모든 노드에 한국어 라벨
6. 관계선에 동사형 라벨 (소모한다, 생성한다, 참조한다 등)
7. 범례 포함 필수

### 노드 수 관리

- 메인 구조도: 최대 15개 노드
- 초과 시: 하위 구조도로 분리하고 메인에 `[[서브시스템명]]`으로 표시
- 각 서브 구조도도 15개 노드 이하

### 검증 기준 (에이전트 자동 검증)
- [ ] Mermaid 문법이 올바르다 (렌더링 가능)
- [ ] 노드 수가 15개 이하이다
- [ ] 모든 노드에 한국어 라벨이 있다
- [ ] 관계선에 라벨이 있다
- [ ] 범례가 포함되어 있다

## Step 6: 플로우차트 작성

> **담당**: system-analyst 에이전트

### 목적
유저의 행동 흐름과 시스템 반응을 단계별로 시각화한다.

### 작성 규칙

1. **메인 플로우 우선**: Happy Path를 먼저 완성
2. **조건 분기 추가**: 성공/실패/부족 등 조건별 경로
3. **예외 경로**: 점선(-.->)으로 표시
4. **서버/클라이언트 구분**: subgraph 또는 색상으로 구분

### 필수 포함 요소

- 시작점과 종료점 (둥근 사각형)
- 유저 액션 (평행사변형, 초록)
- 시스템 처리 (사각형, 파랑)
- 조건 분기 (다이아몬드, 주황)
- 모든 조건에 Yes/No 또는 구체적 조건 라벨

### 플로우 분리 기준

단일 플로우차트가 복잡해지는 경우:
1. 노드 15개 초과 → 서브 플로우로 분리
2. 조건 분기 3단계 이상 중첩 → 분기점별 분리
3. 독립 기능 블록 → 별도 플로우차트

### 검증 기준 (에이전트 자동 검증)
- [ ] 시작점과 종료점이 있다
- [ ] 조건 분기가 1개 이상 있다
- [ ] 모든 경로가 종료점에 도달한다 (데드엔드 없음)
- [ ] 유저 액션과 시스템 처리가 시각적으로 구분된다
- [ ] 노드 수가 15개 이하이다

## Step 7: 상세 명세서 작성

> **담당**: system-analyst 에이전트

### 목적
개발자가 구현에 필요한 UI, 인터랙션, 애니메이션, 사운드 등 세부 사항을 기술한다.

### 작성 규칙

#### UI 레이아웃
- 화면을 상단/중앙/하단, 좌/우 등 영역으로 분해
- 각 영역의 구성 요소, 크기(비율), 정렬 방식 기술
- 스크롤/탭 등 네비게이션 방식 명시

#### 인터랙션
- 모든 터치/클릭 가능 요소의 동작 정의
- 입력 방식 (탭, 더블탭, 롱프레스, 스와이프, 드래그) 명시
- 피드백 (시각적 변화, 진동 등) 명시

#### 애니메이션
- 트리거 조건 (언제 재생되는가?)
- 재생 시간 (초 단위)
- 이징 함수 (ease-in, ease-out 등) - 가능한 경우
- 반복 여부

#### 사운드
- 효과음 트리거 조건
- BGM 변경 트리거
- 볼륨 레벨 (상대적)

## Step 8: 데이터 테이블 설계 & 수치 분석

> **담당**: data-analyst 에이전트
> **참조**: `assets/templates/numerical-analysis-template.md`, numerical-analysis-guide.md

### 목적
시스템 구동에 필요한 데이터 구조를 정의하여 DB 설계의 기초를 제공한다.
data-analyst가 수치 분석을 병행하여 성장 곡선, 전투 공식, 경제 순환, 확률 시뮬레이션을 수행한다.

### 작성 규칙

1. **테이블 목록**: 필요한 모든 테이블을 나열
2. **컬럼 정의**: 각 테이블의 컬럼에 이름, 타입, 설명, 제약조건
3. **샘플 데이터**: 3~5행의 실제 게임 데이터
4. **ER 다이어그램**: 테이블 간 관계 (Mermaid erDiagram)
5. **밸런싱 공식**: 레벨별/등급별 변화 공식이 있다면 수식으로 표기

### 컬럼 타입 규칙

| 타입 | 용도 | 예시 |
|------|------|------|
| INT | 정수값 | 레벨, 개수, ID |
| FLOAT | 소수값 | 확률, 배율 |
| STRING | 문자열 | 이름, 설명 |
| ENUM | 열거형 | 등급(SSR/SR/R), 상태 |
| BOOL | 참/거짓 | 활성 여부 |
| DATETIME | 시간 | 생성일, 만료일 |
| JSON | 복합 데이터 | 보상 목록, 조건 목록 |

### 밸런싱 공식 표기

```
경험치(Lv) = BasExp × (1 + Lv × GrowthRate)^Lv
- BaseExp: 기본 경험치 (100)
- GrowthRate: 성장 계수 (0.15)
```

### 검증 기준 (에이전트 자동 검증)
- [ ] 테이블이 1개 이상 존재한다
- [ ] 모든 컬럼에 이름, 타입, 설명이 있다
- [ ] 샘플 데이터가 3행 이상 있다
- [ ] 테이블 간 관계가 표현되어 있다 (FK 또는 ER 다이어그램)
- [ ] 수치 변화가 있는 데이터에 공식 또는 테이블이 있다
- [ ] 밸런싱 공식이 있으면 xychart-beta 포함
- [ ] 확률 데이터가 있으면 pie 포함
- [ ] 수치 분석 4대 영역 중 해당 영역이 1개 이상 분석되었다
- [ ] 분석에 Chart.js 시각화가 포함되어 있다
- [ ] 수치 분석 결과가 numerical-analysis-template.md 형식으로 산출되었다

## Step 9: 예외 처리 명세

> **담당**: system-analyst 에이전트

### 목적
정상 흐름 외의 모든 예외 상황과 시스템의 대응 방법을 정의한다.

### 분석 영역별 가이드

#### 엣지 케이스
- 재화/재료가 정확히 0일 때
- 인벤토리/슬롯이 꽉 찬 상태
- 동시에 2개 이상의 조건이 변경될 때
- 최초 사용 (데이터가 초기 상태)
- 최대치 도달 상태에서 추가 시도

#### 사이드 이펙트
- 이 시스템의 결과가 다른 시스템에 미치는 영향
- 레벨업 시 자동으로 발생하는 것들
- 재화 차감 시 다른 시스템의 조건 변화
- 아이템 획득/소모 시 도감/업적 연동

#### 동시성 이슈
- 네트워크 지연 중 중복 탭
- 동일 요청의 동시 전송
- 타이머 만료와 유저 액션이 동시에 발생
- 멀티 디바이스 접속 시 충돌

#### 리소스 한계
- 최대 스택/보유 수량
- 레벨/강화 상한
- 시간당/일일 제한
- 인벤토리/저장소 용량

#### 에러 처리
- 서버 응답 없음 (타임아웃)
- 결제 프로세스 실패
- 데이터 불일치 (클라이언트 vs 서버)
- 점검 중 요청

### 표기 형식

```
| 예외 상황 | 발생 조건 | 시스템 반응 | 유저 피드백 | 우선순위 |
|----------|----------|-----------|-----------|---------|
| 재화 부족 | 보유 < 필요 | 요청 거부 | "재화가 부족합니다" + 충전 바로가기 | 높음 |
```

### 검증 기준 (에이전트 자동 검증)
- [ ] 엣지 케이스가 3개 이상 있다
- [ ] 사이드 이펙트가 1개 이상 분석되었다
- [ ] 동시성 이슈가 1개 이상 분석되었다
- [ ] 모든 예외에 "조건 → 시스템 반응 → 유저 피드백" 구조가 있다
- [ ] 우선순위가 지정되어 있다
- [ ] 우선순위 quadrantChart 포함
- [ ] 에러 sequenceDiagram 1개 이상
